# review-stepwise

## Quick start

```
> src/utils/parser.ts をレビューして

[Critical] src/utils/parser.ts:42 — エラーハンドリングなし
parseInput() が例外をスローする可能性があるが、呼び出し元でキャッチされていない。
案: try-catch で囲み、エラーを呼び出し元に伝播させる。
```

ユーザーの判断・指示を受けたら次の項目へ進む。

## ワークフロー

1. レビュー対象を確認（ファイルパス、Globパターン、ブランチ差分など）
2. 改善項目を重要度順（Critical → Major → Minor）に内部的に整理する（ユーザーには提示しない）
3. 改善項目がなければその旨を伝えて終了
4. 最初の項目を提示
5. ユーザーの判断・指示を待つ
6. 対応後、次の項目に進む（ステップ4へ）
7. すべての項目を完了したら終了

## 項目の提示形式

```
[重要度] ファイルパス:行番号 — タイトル
理由を1〜2文で説明
案: 修正方針を1〜2文で提案
```

## 重要度レベル

* Critical: セキュリティ上の問題、致命的なバグ、実装エラー
* Major: アーキテクチャ・デザイン上の改善、パフォーマンス問題、保守性の低下
* Minor: コードスタイルの統一、命名改善、ドキュメントの明確化

## 評価基準

* コードレビュー: coding-standards スキルの基準で評価する
* ドキュメントレビュー: documentation-writing スキルの基準で評価する
* 修正の必要がない項目には言及しない
